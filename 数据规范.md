# 数据规范

本仓库采用“每个实体一个 Markdown 文件”的轻量模型。实体页由两部分组成：顶部 YAML front-matter 存结构化字段，正文写分析、图表、时间线和判断。任何结构化字段不得在正文重复维护。

## 通用约定

| 约定 | 说明 |
| --- | --- |
| 文件命名 | 英文小写短名，单词用 `-` 连接，例如 `waymo.md`、`robotaxi.md`。 |
| 状态 | 必填，枚举建议：`活跃`、`已退出`、`已被收购`、`已上市`。 |
| 数据更新 | 必填，格式为 `YYYY-MM`。 |
| 未核实数值 | 一律填 `~`，不要凭印象补数字。 |
| 定量字段 | 字段名必须带单位后缀，例如 `营收_万元`、`年销量_万辆`。 |
| 来源 | 财务、销量、估值、车队规模等可变数据必须有来源 URL。 |
| 关系字段 | 使用实体名互相点名，不创建数据库或额外关系表。 |

## 垂类评测角度与打分方法

本仓库没有把《赛博汽车》写成一个“官方统一评分表”。公开文章里更常见的是实测、横评和访谈中的观察角度：例如 Robotaxi 体验稿关注“靠不靠谱、是否顺畅、是否安全”；量产 L2 横评关注 ACC 跟车、车道保持、弯道组合控制、紧急避险、目标识别切换、驾驶员注意力监测；无人配送访谈关注无图/端到端、远程脱困、全流程自动化和公众/监管容忍度。

因此，本仓库的 `评价要求` 列实际展示为“赛博汽车评测角度”：它提取文章中的体验和安全观察点，再转成仓库内部 0-100 分制，方便横向比较。该分数不是《赛博汽车》官方分数，也不是监管认证。

通用红线：赛博汽车原文若指出“只是示范”“规模不等于成熟”“合规身份不清”“安全责任链路不完整”“体验还不像日常服务”等问题，对应赛道不得进入 A 档。没有找到对应实测/访谈依据的维度，必须填 `~` 或在来源中标注为待补。

基础角度库：

| 角度 | 适用垂类 | 观察点 |
| --- | --- | --- |
| 座舱/乘坐体验 | Robotaxi、Robobus、Robovan、量产辅助驾驶 | 叫车/上车/下车、车内提示、乘坐舒适性、交互清晰度、乘客信任感。 |
| 行驶安全 | 全部垂类 | 遵守交规/园区规则、碰撞风险、紧急制动、重载制动、弱势交通参与者避让。 |
| 行驶平顺性 | Robotaxi、Robobus、量产辅助驾驶 | 起步、刹车、变道、跟车、弯道控制是否突兀。 |
| 智驾能力 | 量产辅助驾驶、Robotaxi、Robotruck | ACC 跟车、车道保持、弯道组合控制、目标切换、自动变道/NOA、ODD 边界。 |
| 异常接管/远程监管 | L4 垂类 | 远程接管频率、脱困能力、应急流程、责任链路。 |
| 全流程自动化 | Robotruck、Robovan、末端配送、港口、矿区 | 装载、行驶、交接、调度、补能、运维是否形成闭环。 |
| 合规与公众接受 | 开放/半封闭道路垂类 | 路权、车辆身份、数据合规、投诉率、监管容忍度。 |

分档建议：

| 档位 | 分数 | 含义 |
| --- | --- | --- |
| S | 85-100 | 多个实测角度表现稳定，安全、体验、接管和运营闭环均有规模化证据。 |
| A | 75-84 | 主要评测角度表现成熟，但仍有少数场景、合规或远程运营短板。 |
| B | 60-74 | 核心能力可用，仍处于从试点体验到稳定复制的过程中。 |
| C | 0-59 | 体验仍像示范项目，或安全/接管/合规边界不清。 |

## 7 场景赛道分类

按“运营环境（封闭到开放）× 速度”组织：

| 赛道 | 运营环境 | 速度/任务特征 | 典型商业化抓手 |
| --- | --- | --- | --- |
| Robotaxi | 开放道路 | 城市载客，中低速到城市快速路 | 去安全员后的单位经济模型。 |
| 量产辅助驾驶（私家车 L2+/L3） | 开放道路 | 私家车高频通勤与长途 | 前装渗透率、软件订阅、数据闭环。 |
| Robotruck（干线物流） | 开放道路/半封闭 | 高速干线、长距离重载 | 司机成本、燃油/电耗、安全冗余。 |
| Robobus/Robovan（接驳） | 半封闭 | 园区、机场、景区、城市支线 | 固定路线、低速、运营许可。 |
| 末端配送 | 半封闭/开放道路 | 低速、小载荷、高频短途 | 单均配送成本、人力替代。 |
| 港口 | 封闭 | 低速重载、规则场景 | 港区调度系统、无人集卡、吞吐效率。 |
| 矿区 | 封闭 | 低速重载、恶劣工况 | 安全、连续作业、车队调度。 |
